# 메시지 큐

## 메시지 큐의 장점

- Asynchronous

- Decoupling: 어플리케이션과 분리

- Resilience: 일부가 실패시 전체가 영향받지 않음

- Redundancy: 실패할 경우 재실행 가능

- Guarantees: 작업 처리 확인

- Scalable: 다수의 프로세스가 큐 사용

## Apache ActiveMQ

- JMS를 지원하는 클라이언트를 포함하는 Broker, Java뿐만 아니라 다양한 언어를 이용하는 시스템간의 통신 가능

- 클라이언트간 메시지를 송수신할 수 있는 open source Broker

## RabbitMQ

- AMQP 프로토콜 기반 Broker

- 플러그인을 통해 기능 확장 가능

- RabbitMQ Broker는 네트워크 또는 서버 장애 시 신뢰, 배포 및 구성 가능

- Consumer 중심 설계

- JMS는 API, AMQP는 프로토콜. JMS 자바 기반 따라서 다른 언어에서는 사용 못함.

- 오픈 소스에 기반한 표준 프로토콜이 AMQP 그중의 하나의 제품이 RabbitMQ

## Apache Kafka

- 분산 로그를 기반

- 여러 서버에 분산 및 클러스터링하여 가용성 높임

- Producer 중심의 설계

- 구독 방식의 비동기식 구성

## RabbitMQ vs Kafka

- RabbitMQ : 메시지 Broker 방식

- Kafka : pub/sub 방식

- 신속한 요청-응답이 필요한 웹서버 : RabbitMQ

- 백그라운드 작업, 장시간 실행되는 Task, 메시지 통신용

- 데이터 분석, 트래킹, 통합, 로깅, 보안 등을 요구할 때는 Kafka, 실시간 처리

## Kafka

### Kafka 핵심 기능

1.  이벤트 스트림을 발행하고 구독하며, 다른 시스템에 있는 데이터를 끊임없이 import/export 할 수 있음

2.  이벤트 스스팀을 원하는 기간 동안 안전하고 신뢰할 수 있는 곳에 저장

3.  이벤트 발생 즉시 스트림을 처리하거나 모아서 처리

4.  분산 서비스 가능, 확장성 뛰어남, 탄력적이로 내결함성이 있는 안전항 방식으로 제공

5.  다양한 시스템에 배포 가능

6.  고성은 TCP 네트워크 프로토콜로 통신하는 서버와 클라이언트로 구성된 분산 시스템

7.  네트워크 문제나 장비 장애가 발생하더라고 내결함성 유지

### 주요 개념

- Producer : 이벤트를 발행하는 어플리케이션

- Consumer: 이벤트를 구독하는 어플리케이션

- 이벤트는 Topic 으로 구성되어 안전하게 저장

- Topic은 파일 시스템의 폴더와 유사하며, 이벤트는 이 폴더 안에 파일로 비유됨

- 기본 메시징 시스템과 달리 이벤트를 consume 한 후 삭제하지 않음 → 대신 Topic 별 설정으로 이벤트를 보존할 기간 저장, 오래된 이벤트를 폐시

- Kafka의 성능은 데이터의 크기가 크더라도 일정 → 장기간 데이터를 저장해도 문제 없음

- Topic 은 파티셔닝됨 → Topic은 서로 다른 kafka Broker에 있는 여러 bucket으로 분산됨. 데이터를 분산해 저장하는 것은 확정성을 위한 것임

- 클라이언트 어플리케이션은 동시에 여러 Broker 에서 데이터를 읽고 쓸 수 있음.

- 이벤트 키가 동일한 이벤트는 같은 파티션에 기록되며, Kafka 에서는 하나의 Topic-partition 을 consume하는 모든 Consumer는 항상 기록한 순서와 정확히 같은 순서로 해당 partition 이벤트를 읽어간다는 것을 보장

- 내결함성, 고가용성 있는 데이터를 만들기 위해 모든 Topin은 복제할 수 있으며, 지리적으로 분석 혹은 데이터 센터에 걸쳐서 복제함.

- 장애나 Broker 유지 관리 등을 위해 항상 데이터 복제본을 가진 Multi Broker가 존재, 보통 Production 환경에서 replication factor를 3으로 설정

- 데이터 복제는 Topic-partition 레벨에서 이루어짐

### Kafka Apis

- 관리 작업, admin 작업을 위한 command line tool, Java와 Scala를 위한 핵심 API 5rkwl wprhd

  - Topic, Broker나 다른 Kafka 오브젝트를 관리하고 점검 : Admin API

  - 하나 이상의 Kafka Topic에 이벤트 스트림을 write할 수 있는 Producer API

  - 하나 이상의 Topic을 구독하고 해당 Topic에 생성한 이벤트 스트림을 처리하는 Consumer API

  - 스트림 처리 어플리케이션과 마이크로 서비스를 구현할 수 있는 Kafka Strems API : 데이터 변환, 집계, Join, Windowing, 이벤트 시간 기반 처리, 이벤트 스트림 처리 등

  - 재사용 가능한 데이터 import/ export 커넥터를 만들고 실행할 수 있는 Kafka Connector API

### Kafka 사용 사례

- Messaging : 전통적인 메시지 Broker 대체

- Website Activity Tracking: 사용자 활동 추적

- Metrics: 운영 모니터링 데이터. 분산 어플리케이션의 통계 데이터 집계

- Log Aggregation: 로그 집계 솔루션 대용

- Stream Processing: 데이터 처리 파이프라인 Kafka Streams

- Event Sourcing

- Commit Log : 분산 시스템에서 외부 로그 커밋, 여리서 로그는 데이터 자체를 의미

## quick-start

1.  다운 및 압축 풀기

    1.  tar -xzf kafka_2.13-2.7.0.tgz

    2.  cd kafka_2.13-2.7.0

2.  Kafka 시작

    1.  java 8이상 설치

    2.  bin/zookeeper-server.start.sh config/zookeeper.properties

    3.  터미널 열고 실행 bin/kafka/server-start.sh config/server.properties

3.  Topic 생성

    1.  이벤트 생성 순서. 토픽 생성 →

    2.  bin/kafka-topics.sh --create --topic quickstart-events --bootstrap-server localhost:9092

4.  Topic 안에 event 생성

    1.  bin/kafka-console-producer.sh --topic quickstart-events --bootstrap-server localhost:9092

    2.  This is my first event

    3.  This is my second evnet

    4.  Ctrl-C 로 중지

5.  이벤트 읽기

    1.  터미날 하나 더 열기

    2.  bin/kafka-console-consumer.sh --topic quickstart-events --from-beginning --bootstrap-server localhost:9092

    3.  producer에 write한 이벤트 출력됨

    4.  Ctrl-C 로 종료

6.  Kafka Connect로 이벤트 스트림을 import/export 하기

    1.  Kafka Connector로 DB나 메시징 시스템에 연결하여 처리 가능 - Kafka Connect secion 참조

7.  Kafka Streams로 이벤트 처리하기

    1.  Kafka Streams Client Library 를 사용하여 데이터 처리

    2.  샘플

  @EnableKafka
  @Configuration

    @Value(value="${spring.kafka.bootstrap-servers}")

    public KafkaAdmin kafkaAdmin() {
      Map<String, Object> configs = new HashMap<>();
      configs.put(AdminClientConfig.BOOTSTRAP_SERVERS_CONFIG, bootstrapAddress);

    public NewTopic myTopic1() {

    @Bean
    public NewTopic myTopic2() {
      return new NewTopic("my_topic_2", 1, (short)1);
    }

    @Bean
    public ProducerFactory<String, String> producerFactory() {
      Map<String, Object> configProps = new HashMap<>();
      configProps.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, bootstrapAddress);
      configProps.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
      configProps.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
      return new DefaultKafkaProducerFactory>(configProps);
    }

    @Bean
    public KafkaTemplate<String, String> kafkaTemplate() {
      return new KafkaTemplate<>(producerFactory());
    }

    @Bean
    public ConsumerFactory<String, String> consumerFactory() {
      Map<String, Object> props = new HashMap<>();
      props.put(ConsumerConfig.BOOTSTRAP_SERVERS_CONFIG, bootstrapAddress);
      props.put(ConsumerConfig.GROUP_ID_CONFIG, "foo");
      props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringSerializer.class);
      configProps.put(ConsumerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
      return new DefaultKafkaConsumerFactory>(props);
    }

    @Bean
    public ConcurrentKafkaListenrContainerFactory<String, String> kafkaListenerContainerFactory() {
      ConfurrentKafkaListenerContainerFactory<String, String> factory = new ConcurrentKafkaListenerContainerFactory<>();
      factory.setConsumerFactory(consumerFactory());
      return factory;
    }
  }
