### 속성

#### propagation(트랜잭션 전파)

- 트랜잭션을 시작하거나 기존 트랜잭션에 참여하는 방법을 결정하는 속성

- 선언적 트랜잭션 경계 설정 방식의 장점은 여러 트랜잭션 적용 범위를 묶어서 커다란 트랜잭션 경계를 만들 수 있음 (트랜잭션을 조합해서 새로운 트랜잭션을 만들거나 별개로 만들 수 있다는 말?)

- Spring이 지원하는 트랜잭션 전파 속성은 여섯가지가 있음

- org.springframework.transaction.annotation.Propagation에 정의된 enum 값 사용

  - REQUIRED : 디폴트 속성, 미리 시작된 트랜잭션이 있으면 참여하고 없으면 새로 시작

  - SUPPORTS : 이미 시작된 트랜잭션이 있으면 참여하고 그렇지 않으면 트랜잭션 없이 진행. <u>트랜잭션이 없긴 하지만 해당 경계 안에서 Connection이나 하이버네이트 Session 등을 공유할 수 있다.</u>

  - MANDATORY : 이미 시작된 트랜잭션이 있으면 참여, 트랜잭션이 시작된 것이 없으면 예외 발생. 혼자서 독립적으로 트랜잭션을 진행하면 안 되는 경우 사용

  - REQUIRES_NEW : 항상 새로운 트랜잭션 시작, 이미 진행 중인 트랜잭션이 있으면 트랜잭션 잠시 보류.

  - NOT_SUPPORTED : 트랜잭션을 사용하지 않게 한다. 이미 진행 중인 트랜잭션이 있으면 보류

  - NEVER : 트랜잭션을 사용하지 않도록 강제

  - NESTED : 이미 진행중인 트랜잭션이 있으면 중첩 트랜잭션을 시작. 부모 트랜잭션에 영향을 미치지 않는 자식 트랜잭션을 실행. JDBC3.0 스펙의 SavePoint를 지원하는 드라이버와 DataSourceTransactionManager 를 이용할 경우에 적용 가능. 적용 가능한지 여부를 확인하고 사용해야함

#### isolation(트랜잭션 격리수준)

- 동시에 여러 트랜잭션이 진행될 때에 트랜잭션의 작업 결과(or 처리하는 데이타)를 다른 트랜잭션에게 어떻게 노출할 것인지 결정하는 기준

  - DEFAULT : 사용하는 데이터 엑세스 기술 또는 DB 드라이버의 디폴트 설정에 따름.

  - READ_UNCIMMITTED : 가장 낮은 격리 수준. 하나의 트랜잭션이 commit이 되기 전에 그 변화가 다른 트랜잭션에 그대로 노출되는 문제.

  - READ_COMMITTED : 가장 많이 사용되는 격리 수준. 다른 트랜잭션이 커밋하지 않은 정보는 읽을 수 없음.

  - REPEATABLE_READ : 하나의 트랜잭션이 읽은 로우를 다른 트랜잭션이 수정하는 것을 막아준다. 새로운 로우를 추가하는 것은 제한하지 않는다.

  - SERIALIZABLE : 가장 강력한 트랜잭션 격리 수준. 트랜잭션을 순차적으로 진행시켜주기 때문에 여러 트랜잭션이 동시에 같은 테이블의 정보를 액세스하지 못한다. 성능이 떨어짐

#### timeout(제한시간)

- 트랜잭션 시스템의 제한 시간 정의

#### read-only,readOnly(읽기전용 트랜잭션)

- 트랜잭션을 읽기 전용으로 설정

- INSERT, UPDATE, DELETE 작업 발생이 예외 발생

#### rollback-for, rollbackFor, rollbackForClassName(트랜잭션 롤백 예외)

- 선언적 트랜잭션에서는 런타임 예외가 발생하면 롤백

- 스프링에서는 데이터 액세스 기술의 예외는 런타임 에외로 전환돼서 던져지므로 런타임 예외를 롤백 대상으로 삼음

- 체크 예외(일반적으로 커밋대상)지만 롤백 대상으로 삼아야 하는 경우 rollback-for 나 annotaion의 rollbackFor 를 정의하여 예외를 지정

- xml 예제

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  <tx:method name="get*" read-only="true" rollback-for="NoSuchMemberException">
  ```

  </div>

  </div>

- annotation 예제

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  @Trasactional(readOnly=true, rollbackFor=NoSuchMemberException.class)
  ```

  </div>

  </div>

#### no-rollback-for, noRollbackFor, noRollbackForClassName(트랜잭션 커밋 예외)

- rollback-for 속성과는 반대로 롤백 대상인 런타임 예외를 트랜잭션 커밋대상으로 지정

세밀한 트랜잭션 속성 지정을 할 때는 @Transactional 를 사용하는데 일반적으로 세밀한 트랜잭션 설정보다는 xml 로 메소드 이름 패턴을 이용해서 트랜잭션 설정을 함.

## 데이터 액세스 기술 트랜잭션의 통합

- 서로 다른 데이터 액세스 기술을 사용할 때의 트랜잭션 통합에 관한 사용법

  - JPA DAO 와 JDBC DAO 를 같이 사용할 때 트랜잭션을 어떻게 통합해서 사용할 것인지

  - JDBC 와 myBatis는 등등

- 기본 원칙은 DB 당 트랜잭션 매니저는 하나만 사용

### 트랜잭션 매니저별 조합 가능 기술

#### DataSourceTransactionManager

- JDBC와 iBatis (myBatis 도 가능) 두가지 기술을 함께 사용할 수 있음

- JDBC DAO와 iBatis DAO가 같은 DataSource를 사용하도록 만들어주면 됨

- page 331 그림 참조

#### JpaTransactionManager

- JPA의 트랜잭션은 JPA API를 이용해 처리됨

- Spring 에서 JPA의 EntityManagerFactory 가 Spring bean으로 등록된 DataSource를 사용할 수 있음

- JPA에서 DataSource를 사용하게 하면, 동일한 DataSoure를 JDBC DAO나 iBatis DAO에서 사용하도록 하여 하나의 트랜잭션 매니저로 관리 가능

- page 332 그림 참조

#### HibernateTransactionManager

- HibarnateDAO를 사용하려면 등록

- 다른 방식과 동일하게 DataSource를 공유하도록 설정하여 사용

- SessionFactory가 DataSource를 사용하도록 설정

#### JtaTransactionManager

- 서버가 제공하는 트랜잭션 서비스를 JTA를 통해 이용하면 모든 종류의 데이터 액세스 기술의 DAO가 같은 트랜잭션 안에서 동작가능

- JTA 서버 환경 필요, XA를 지원하는 DataSource 구성 필요

### ORM과 비 ORM DAO를 함께 사용할 때의 주의 사항

- JPA, Hibarnate 같은 ORM 기술과 JDBC, iBatis 같은 비 ORM을 사용할 때
