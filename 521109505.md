## 멀티 스레드 이슈

- 여러 개의 CPU가 존재할 시, RAM에 있는 데이터를 각 CPU의 Cache Memoty로 복제하여 사용, CPU가 처리한 데이터를 Cache Memory 에서 RAM 으로 덮어 쓰는 과정에서 RAM에 존재하는 동일한 데이터 에 대해 각 CPU Cache MEmory의 데이터가 다를 수 있음 - 가시성, 동시접근 문제 발생

- 가시성

- 동시 접근 문제

## 가시성 문제

- 하나의 Thread 에서 공유 자원(변수, 객체 등)을 수정한 결과가 다른 Thread에 보이지 않을 경우 발생

- 샘플 예제

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  public class VisibleIssue {
    private static boolean stopRequested;

    public static void main(String[] args) throws InterruptedException {
      Thread background = new Thread(() -> {
        for ( int i = 0; !stopRequested; i++);
        System.out.println("background 스레드가 종료");
      });

      background.start();

      TimeUnit.SECONDS.sleep(1);
      stopRequested = true;
      System.out.println("main 스레드가 종료");
    }
  }
  ```

  </div>

  </div>

  - 출력 결과 : main 스레드가 종료

  - stopRequested = true 설정을 background가 인지 못함

  - 같은 CPU가 아닌 경우, background Thread와 main Thread가 RAM 상의 데이터를 cache memory에 복제하여 데이터 사용

  - main thread가 변경한 cache memory 상의 데이터를 background thread가 같은 cpu 내 cache memory가 아니므로 볼 수 없음 → 가시성 문제 발생

## 동시 접근 문제

- 여러 Thread 에서 공유 자원(변수, 객체 등)을 동시 접근하였을 떄, 연산이 가장 늦게 끝난 결과값으로 덮어 씌워지는 문제

- 샘플 예제

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  public class ConcurrentIssue {
    private static int t;

    public static void main(String[] args) throws InterruptedException {
      for ( int i = 0; i < 100; i++) {
        new Thread(() -> {
          for (int j = 0; j < 1000; j++)
            System.out.println(t++);
        }).start();
      }
    }
  }
  ```

  </div>

  </div>

  - 예상값을 100 \* 1000 = 100000 이어야 할 것 같지만 각 cpu 에서 수행된 thread는 cache memory의 값을 RAM에 저장하는 시점과 RAM 에서 cache memory 로 복제하여 사용하는 시점에 따라 값이 틀려짐

## 동시성의 해결책

- synchronized

- volatile

  - 키워드가 붙은 자원은 하나의 Thread만 write 하고 다른 Thread는 read 만 한다는 전제하에서 동시성 보장

  - volatile keyword가 붙은 자원은 read/write 작업이 CPU의 cache memory 가 아닌 Main Memory 에서 이루어짐

  - 사용하는 방법에 따라 여러 Thread가 Main Memory 상의 데이터에 접근 write 가능한가 체크함 → 가시성 해결, 동시설 해결 못함

- Atomic 클래스 사용(CAS compare-and-swap 정책)

  - synchronized 보다 적은 비용으로 동시성 보장

  - CAS 알고리즘: 현재 Thread가 존재하는 CPU의 cache memory와 Main memory에 저장된 값을 비교하여 일치하는 경우, 새로운 값으로 교체하고, 일치하지 않은 경우 기존 교체가 실패되고, 이에 대해 계속 재시도하는 방식 - 연산 전의 값을 가지고 있다가 비교한다는 말

  - CPU가 Main Memory 자원을 Cache Memory로 복제해와 연산을 수행하는 동안 다른 Thread 에서 연산이 수행되어 Main Memory의 자원 값이 바뀌었을 떄 기존 연산을 실패 처리하고 새로 바뀐 Main Memory 값을 다시 복제해와서 재수행
