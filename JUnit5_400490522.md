### @TestConfiguration

- Test 시 Database나 DI 된 클래스를 대체하기 위해 기존에 정의된 Configuration이 아닌 Test 시 사용할 Configuration을 별도로 구현한 후, 클래스 명 위에 @TestConfiguration 정의

- 해당 Configuration을 사용할 Class 에서 @Import(Configuraion.clss) 을 정의하여 사용

# Mokito와 JUnit 같이 사용하기

## 용어

- test doubles(mocks) : 테스트 목적으로 프로덕션 객체를 대체하는 모든 경우에 대한 일반적인 용어

- mockito에는 다음과 같은 test doubles를 작업한다.

  - stubs : 테스트 도중 실행된 메소드에 대해 사전에 정의된 값이 있는 객체

  - spies : Stubs와 유사, 실행하는 방법에 대한 통계를 기록

  - Mocks : 테스트 도중 실행된 메소드에 대한 반환값과 그에 대한 기대값이 정의된 객체. 예상하지 못했던 결과에 대해 Exception을 발생시킴

- interface와 classes에 대해 mock할 수 있음

- mockito annotation을 사용하면 최소한의 상용구 코드를 생성하는 데 도움이 됨

- mock이 생성되면 모든 상호작용에 대해 기억하고, 상호작용 중에 흥미 있는 것이 무엇이든 간에 선택적으로 확인 가능

## Mock 메소드

### verify()

- mock 객체에 대해 원하는 메소드가 특정 조건으로 시행되었는지 검증

- atLeastOnce() : 적어도 한 번 수행했는지 검증

- atLeast(int n) : 적어도 n 번 수행했는지 검증

- times(int n) : 무조건 n 번 수행했는지 검증 (n보다 크거나 작으면 오류로 간주)

- atMost(int n) : 최대한 n번 수행했는지 검증

- never() : 수행되지 않았는지 검증(수행했으면 오류)

### when()

- 원하는 값을 리턴하는 기능

- when() 메소드 호출시 OngoingStubbing interface 리턴

- thenAnswer(Answer\<?\> answer) : Answer interfae를 구현하여 원하는 작업을 수행

- thenCallRealMethod() : 해당 메소드가 구현되어 있다면 실제 메소드를 호출

- thenReturn(T value) : 지정된 값을 리턴

- thenReturn(T Value, T… Value) : 지정되어 있는 값을 순차적으로 리턴

- thenThrow(java.langThrowable… throwables) : 예외를 야기시키는 Throwable 객체를 지정

# @Mock과 @Spy차이점

- @Mock을 사용할 때 Mockito는 필드 유형에 대한 기본적인 shell 인스턴스를 생성하여 전적으로 그것과의 상호작용을 추적하도록 계측되어진다.

- 그 것은 실제 객체가 아니며 그것에 대한 상태 변경을 유지하지 않는다.

- @Spy 를 사용할 때 Mockito는 클래스의 실제 인스턴스를 생성하고 그것과의 모든 상호작용을 추적한다. 상태 변경도 유지한다.

## void 메소드를 mock하는 방법

1.  doNothing() : void 메소드를 호출하는 것을 완전히 무시하는 것 : 기본 행동

2.  doAnswer() : void 메소드를 호출 될 때 실행 시간, 복잡한 오퍼레이션 구성

3.  doThrow() : 모의 void 메소드 호출될 대 exception 던지기

4.  doCallRealMethod() : 모의 객체를 부르지 않도록 실제 메소드 호출

## doNothing()

- void method call을 완전히 무시할 때 사용

  UserRepository mockedUserRepository;
