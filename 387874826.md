# Spring Boot

- Spring : Spring Framework

- Boot : BootStrap

- Spring Application 을 bootstrapping 하는 데 도움이 되는 오픈소스 Java 기반 프레임워크

- Spring.io의 공식 정의 : Spring Boot을 사용하면 독립 실행형 프로적션 급 Spring 기반 애플리케이션을 쉽게 만들 수 있음

- Spring Boot 는 Spring 기반 애플리케이션을 생성하는 프로세스를 단순화

- Spring Boot가 생성하는 Spring 기반 애플리케이션의 jar/war 생성 및 웹 서버 배포 등은 Spring Boot 에서 처리

## Spring Boot 가 유명한 이유

1.  프로젝트 개발 속도를 높임

2.  웹 기반 및 간단한 애플리케이션을 구성, 설정, 실행하는 쉽고 빠른 방법 제공

3.  별도로 구성한 웹서버 필요하지 않음. 그러므로 war 파일을 수동으로 배포할 필요 없음

4.  XML 설정을 줄이고 annotation 사용을 촉진

5.  Spring Framework의 하나 이상의 annotatoin을 결합하고 그 결합을 하나의 annotation으로 대체함. 예를 들어 @SpringBootApplication 은 @EnableAutoConfiguratoin, @Configuratoin, @ComponentScan annotation의 조합임

6.  엔터프라이즈급 애플리케이션을 제공하기 위한 개발자의 노력을 줄임

7.  모든 프로젝트에 일반적이로 적용 가능한 ready-made(사전에 준비된, 기성품) 기능 제공. 따라서 boiler plate code 를 제거(모든 코드를 작성하기 위해 항상 필요한 부분 - 별 수정없이 반복적으로 사용되는 코드란 의미)

8.  jar 파일을 사용하여 실행할 수 있는 독립 실행 애플리케이션 생성

9.  Spring 과 타사 라이브러리를 자동으로 설정, 구성함

10. 헬스 체크, 메트릭, 외부화된 구성과 같은 production-ready 기능을 제공

11. 많은 타사 도구와 쉽게 통합할 수 있는 매커니즘 제공

12. Java 에서 마이크로서비스를 개발할 수 있도록 훌륭한 지원을 제공

13. Spring Boot는 Spring JDBC, Spring ORM, Spring DATA, Spring Security, Spring Web MVC와 같은 생태계와 쉽게 통합됨

14. 데이터 베이스와 쉽게 통합됨. 때때로 driver, dialects 등 데이터 베이스 관련 속성을 제공할 필요 없음. 데이터 베이스 URL에서 자동으로 이러한 속성 값을 가져옴

15. 초기 구성에 시간을 낭비 하지 않도록 기능을 빠르게 작업할 수 있는 ready-made 시작 프로젝트를 제공

16. 애플리케이션을 빠르게 개발하고 테스트할 수 있도록 CLI(Command Line Interface) 제공

17. Maven과 Gradle 같은 빌드 도구 지원

# Spring Boot 장점

1.  Spring Boot Starters

    - 개발 프로세스를 단순화할 수 있는 mini starter projecct 제공

    - Spring Web, Spring Data JPA, Spring Security, Spring Boot Dev Tools, MySQL Driver 등

    - Lombok과 같은 타사 라이브러리를 starter 프로젝트 형태로 추가할 수 있는 유연성 제공

2.  Boilerplate Code 감소

    - boilerplate code, annotations, 복잡한 XML 구성 작성하는 수동 작업을 피할 수 있게 도와줌

    - boilerplate code를 반복해서 작성하지 않아도 되는 쉽게 사용할 수 있는 API를 제공

    - 여러 annotation 조합을 하나의 annotation으로 대체할 수 있는 기능도 제공

3.  Embedded Servers

    - Tomcat, Jetty 및 UnderTow 같은 임베디드 HTTP 서버 제공(jar 형태로)

    - Tomcat은 default로 활성화 되어 있고 다른 서버는 설정을 통해 활성화 가능

4.  쉬운 Database 연결

    - Oracle, PosgreSQL, MySQL, MongoDB, Redis 등과 같은 데이터 베이스와 쉽게 연결 가능

    - ActiveMQ, RabbitMQ, ElasticcSearche 등 같은 대기열 메시징 서비스와 쉽게 통합 가능

5.  Microservices 개발 지원

    - Microservice와 함께 작동 가능

    - Spring Boot artifacts 를 Docker containers에 직접 배포 가능

6.  테스팅 기본 설정

    - Unit 과 통합 테스트를 위한 기본 설정을 제공

7.  Jar 파일로 애플리케이션 실행

    - java -jar 명령을 사용하여 jar를 독립적으로 실행하도록 준비가 되어 있음

8.  Spring Boot Profile 로 쉬운 환경 전환

    - Profile 이름과 서버 세부 정보를 변경하기만 하면 다양한 환경에서 애플리케이션을 실행하는 프로세스를 단순화함

9.  애플리케이션 모니터링을 위한 Spring Boot Actuator

    - Spring Boot Actuator를 통해 애플리케이션의 내부 작업을 검사할 수 있는 기능을 제공

    - Spring Boot Actuator는 Spring Boot 애플리케이션이 관리 및 모니터링을 위한 엔드포인트 제공

10. 생산성 증가 와 개발 시간 감소

# Spring Boot 단점

1.  필요하지 않는 추가 dependency가 많이 포함되어 있음. 따라서 배포 파일의 크기가 늘어남

2.  레거시 Spring 기반 프로젝트의 코드를 업데이트하는데 어려움이 있음. Spring Boot CLI를 사용하면 이 문제를 극복할 수 있음. 레거시 코드를 변환하는데 도움이 됨

# Spring Boot 애플리케이션을 생성하는 방법

1.  Spring Boot CLI Tool 사용

2.  Spring STS IDE 사용 (또는 intellij 등 사용)

3.  Spring Initializr Website 사용

STS(Spring Tool Suite) IDE는 Spring Boot 프로젝트를 개발하는데 가장 일반적으로 사용되는 방법

# Spring Boot 에서 Starters 란

- Starter는 애플리케이션에 포함할 수 있는 미리 만들어진 작은 프로젝트

- 공식적으로는 Dependency descriptors 라 함

- Starter에는 프로젝트를 빠르게 시작하고 실행하고, 일관되고 지원되는 관리된 전이적 종속성 세트를 지원하는 많은 dependencies를 포함하고 있다.

- 모든 공식 starter는 spring-boot-starter-\*, 와 같은 유사한 이름 지정 패턴을 따른다.

# Spring Boot 주의 사항

## default 패키지 사용

- 클래스에 package 선언이 없을 때 해당 클래스는 default package에 있는 것으로 간주

- default package 활용은 일반적으로 권장하지 않음

- Spring Boot Application에서 default package에 @ComponentScan, @ConfigurationPropertiesScan @EntityScan @SpringBootApplicatoin annotation을 적용하게 되면 **모든 jar 에 있는 클래스를 전부 읽음**

## Main Application class 위치

- Main Application class 는 다른 클래스들을 아우르는 Root package에 배치하는 것을 권장

- **@SpringBootApplication** annotation이 정의된 클래스의 package 정보로부터 특정 항목들을 찾기 위한 “검색 패키지”를 암시 : <u>@SpringBootApplication 정의된 클래스의 package부터 componentscan 진행</u>

- @SpringBootApplication 을 사용하고 싶지 않으면 **@EnableAutoConfiguration** 과 **@ComponentScan**을 대신 사용(@SpringBoogApplication이 import)

- @SpringBootApplication 이 정의된 패키지와 별개의 package를 Component Scan 대상에 포함시키려면 annotation 설정에 추가

  ``` syntaxhighlighter-pre
  @SpringBootapplication
  @ComponentScan(basePackages= {"com.whale","net.comlade"})

  }
  ```

## Configuration classes

- Spring Boot 는 자바 기반 설정 선호

- @Configuratoin 단일 클래스로 설정 정의

- @Import 로 다른 설정 클래스 import 가능

- @ComponentScan 으로 자동 스캔 선택도 가능

- @ImportResource 로 XML 설정 파일 로드

## Auto-configuration

- 자동 설정은 **클래스 패스에 추가한 jar 의존성을 기반으로** 자동으로 설정됨 : dependency 에 추가되어 있으면 자동으로 설정된다는 말

- 자동 설정이 실행되는 조건 : @Configuration 클래스 중 하나에 @EnableAutoConfiguration 이나 @SpringBootApplication이 추가되어야 함(Main 외에 Configuration 클래스에 추가?)

- 일반적으로 @Configuration 클래스에 @EnableAutoConfiguration을 추가하겠지..

- 현재 적용되고 있는 자동 설정과 그 이유(원인 클래스 or jar) 를 확인하려면 --debug 옵션으로 application 시작

- 원하지 않는 특정 클래스 자동 설정되지 않도록 제외

  - @SpringBootApplication 의 exclude 속성 사용

## Automatic Restart

- spring-boot-devtools를 사용하는 application은 classpath에 있는 파일이 변경될 때마다 자동으로 재시작됨

- 빌드 플러그인을 사용하여 maven 이나 gradle 로 재시작한다면 반드시 forking을 enabled 로 설정

- 자동 재시작은 <a href="https://godekdls.github.io/Spring%20Boot/developing-with-spring-boot/#683-livereload" class="external-link" rel="nofollow">LiveReload</a>와 함께 사용할 때도 매우 잘 동작함

### Restart Classloader 커스터마이징

- 재시작 기능은 두 개의 클래스 로더를 사용해 구현

  - IDE에 열려 있는 모든 프로젝트는 “restart” 클래스 로더로 로드, 일반 .jar 파일은 “base” 클래스 로더로 로드

- 멀티 모듈 프로젝트 작업시, 모든 모듈을 IDE로 import 하지 않았다면 restart 클래스 로더에 로딩할 모듈을 지정해야 할 수 있음 =\> META-INF/spring-devtools.properties 파일에 정의

  - spring-devtools.properties 에 restart.exclude, restart.include 로 정의

  - restart.exclude “base” 클래스 로더로 가져와야 할 항목

  - restart.include “restart” 클래스 로더로 가져와야 할 항목

  - 정의

    <div class="code panel pdl" style="border-width: 1px;">

    <div class="codeContent panelContent pdl">

    ``` syntaxhighlighter-pre
    restart.exclude.companycommonlibs=/mycorp-common-[\\w\\d-\\.]+\\.jar
    restart.include.projectcommon=/mycorp-myproj-[\\w\\d-\\.]+\\.jar
    ```

    </div>

    </div>

    <div class="code panel pdl" style="border-width: 1px;">

    <div class="codeContent panelContent pdl">

    ``` syntaxhighlighter-pre
    restart:
      exclude:
        companycommonlibs: "/mycorp-common-[\\w\\d-\\.]+\\.jar"
      include:
        projectcommon: "/mycorp-myproj-[\\w\\d-\\.]+\\.jar"
    ```

    </div>

### 알려진 한계사항

- 재시작 기능은 표준 ObjectInputStream을 사용해 deserialize한 Object에서 제대로 동작하지 않는다.

- 데이터를 deserialize 해야 한다면 Spring의 ConfigurableObjectInputStream을 Thread.currentThread().getContextClassLoader()와 함께 사용해야 함

## Global Settings

- \$HOME/.config/spring-boot 디렉토리에 다음 파일 중 하나를 추가하면 global devtools 설정을 구성 가능 : 해당 서버에 있는 모든 Spring Boot Application에 적용됨

  - spring-boot-devtools.properites

  - spring-boot-devtools.yaml

  - spring-boot-devtools.yml

- 이 파일에 추가한 모든 property는 devtools를 사용하는 해당 기기의 모든 spring boot application 에 적용됨

- spring-boot-devtools 는 프로파일 지원 안함

### Configuring File System Watcher

- 특정 시작 간격으로 클래스의 변경 사항을 폴링해서 미리 정의된 휴지 기간(quiet period) 동안 기다리면서 더 이상 변경이 없는지 확인

- Spring Boot는 파일을 컴파일하고 특정 위치로 복사하여 실행하도록 하는데 개발시에는 IDE 에 의존하기 때문에 변경사항이 특정 시간동안 반영이 안 될 수 있기 때문에 제공하는 기능 : 개발시에만 적용됨

- spring.devtools.restart.poll-interval, spring.devtools.restart.quiet-period 를 개발환경에 맞는 값으로 늘려서 설정

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  spring.devtools.restart.poll-interval=2s
  spring.devtools.restart.quiet-period=1s
  ```

  </div>

  </div>

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  spring:
    devtools:
      restart:
        poll-interval: "2s"
        quiet-period: "1s"
  ```
