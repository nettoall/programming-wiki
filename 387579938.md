# Spring Boot 특징

## SpringApplication

### Startup Failure

- SpringApplication 시작시 기본적으로 INFO 로그 메시지 출력

  - 로그 출력을 끄려면 **spring.main.log-startup-info=false** 로 설정

  - 기동시 부가적인 로그를 넣으려면 SpringApplication의 하위 클래스에서 logStartupInfo(boolean) 재정의

- application 기동시 오류는 FailureAnalyzers 가 해결법 및 조치 방법을 알려줌

  - failure analyzer가 없더라도 전체 컨디션 리포트를 표기하면 무엇이 잘못되었는지 확인하기 쉬움

  - org.springframework.boot.autoconfigure.logging.ConditionEvaluationReportLoggingListener 에 **debug 로그 활성화**

    <div class="codeContent panelContent pdl">

    ``` syntaxhighlighter-pre
    java -jar myproject-0.0.1-jSNAPSHOT.jar --debug
    ```

### Lazy Initialization

- 필요에 따라 Application을 Lazy 방식으로 초기화 가능

  - 장점 : 초기 메모리 로딩 사이트 줄이고 기동 속도 빠름

  - 단점 : 어플리케이션에 오류가 있더라도 해당 기능이 최초 호출될 때 알게됨

  - 사용 방법

    - SpringApplicationBuilder lazyInitialization, SpringApplication **setLazyInitialization**을 사용

    - **spring.main.lazy-initialization=true**

- Banner 사용법은 메뉴얼 참고

- SpringApplication customizing

  - 필요한 초기값을 설정하여 기동 가능

  - main()에 정의하거나, properties 에 정의하여 사용 설정

  - 설정 옵션은 SpringApplication api doc 참고

### Application Availabilty

- micro service에서 본 듯, Spring Boot는 가용 상태 “liveness”, “readiness” 제공

- Spring Boot의 Actuator 기능을 사용하면 K8s 같은 플랫폼의 health endpoint 그룹으로 노출됨(상태가 모니터링 된다고 보면 됨)

- 자체 bean에 ApplicationAvailabilty interface를 주입하여 직접 가용 상태 조회 가능

- Liveness State : application의 liveness 상태를 내부 상태가 올바르게 작동할 수 있는 상태인지 또는 현재 실패했다면 자체적으로 복구할 수 있는 상태인지 알려줌

- Readiness State : application이 traffic을 처리할 준비가 되었는지 나타냄

#### Managing the Application Availability State

- Application Component는 ApplicationAvailabilty interface를 주입해서 여기 있는 메소드를 호출하면 현재 가용 상태를 조회 가능

- 쿠버네티스 “exec Probe” 로 상태를 알 수 있도록 Application의 “Readiness” 상태 파일로 export

  ``` syntaxhighlighter-pre
    }
  }
  ```

### Application Events and Listeners

- 어떤 event는 실제로 ApplicationContext가 생성되기 전에 trigger 되므로 이런 이벤트는 @Bean 으로 Listener를 등록할 수 없음

- 이런 이벤트 Listener 는 SpringApplication.addListeners(…) 나 SpringApplicationBuilder.listeners(…) 로 등록

- Application 생성 방식에 관계없이 자동으로 등록되도록 하려면 **META-INF/spring.factories** 파일에 추가

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  org.springframework.context.ApplicationListener=com.example.project.MyListener
  ```

  </div>

  </div>

- Application 실행시 Application Event 전송 순서

  1.  기동 시작, Listener와 Initializer 등록, **ApplicationStartingEvent** 전송

  2.  Context에서 사용할 Environment 파악하면 Context 생성하기 전에 **ApplicationEnvironmentPreparedEvent** 전송

  3.  ApplicationContext가 준비되면 ApplicationContextInitializer 호출, Bean 정의 로딩하기 전 **ApplicationContextInitializedEvent** 전송

  4.  Bean 정의 로드하고 나서 **ApplicationPreparedEvent** 전송

  5.  Context refresh한 후에 Application과 Commandlien runner 호출하기 전에 **ApplicationStartedEvent** 전송

  6.  AvailabilityChangeEvent로 **LivenessState.CORRECT** 전송해 Application 활성 상태임을 알림

  7.  Application과 commandLine runner을 호출한 후에 **ApplicationReadyEvent** 전송

  8.  직후에 AvailabilityChangeEvent로 **ReadinessState.ACCEPTING_TRAFFIC** 전송해 서비스 요청 처리할 준비가 되었음을 알림

  9.  기동시 예외가 있으면 **ApplicationFailedEvent** 전송

- 기타 이벤트

  - WebServer가 준비되고 나면 **WebServerInitializedEvent** 전송

  - Servlet과 Reactive에서 각각 **ServletWebServerInitializedEvent**와 **ReactiveWebServerInitializedEvent** 전송

  - ApplicationContext라 refresh 되면 **ContextRefreshedEvent** 전송

- Spring Boot는 내부적으로 이벤트를 사용해 다양한 Task 처리

- Application event는 Spring Framework의 event publishing mechanism을 사용해 전송

- 이 메커니즘으로 인해 자식 context의 listner에 publish된 event는 모든 상위 context의 listener에서 publish됨

- 따라서 Application이 SpringApplication instance의 계층구조를 사용한다면 listener는 같은 유형의 application event instance를 여러 번 수신 가능

- listener에서 해당 context에 대한 event와 하위 context에 대한 event를 구별하려면 해당 application context 주입을 요청한 다음 주입한 context를 event의 context와 비교

- context는 ApplicationContextAware를 구현하거나 Listener가 Bean 일 때 @Autowired 를 사용해 주입

### Web Environment

- SpringApplication은 사용자를 대신해서 ApplicationContext를 올바른 타입으로 생성

- WebApplicationType을 결정하는데 사용하는 알고리즘은 다음과 같음

  - Spring MVC가 있다면 AnnotationConfigServletWebServerApplicationContext 사용

  - Spring WebFlux가 있다면 AnnotationConfigReactiveWebServerApplicationContext 사용

  - 그외엔 AnnotationConfigApplicationContext 사용

- 하나의 Application에서 Spring MVC와 Spring WebFlux의 새 WebClient를 동시에 사용한다면 기본적으로 Spring MVC 사용

- setWebApplicationType(WebApplicationType) 을 호출하여 간단히 재정의 가능

### Application Exit

- SpringApplication 은 JVM에 shutdown hook을 등록해서 종료시 ApplicationContext가 정상적으로 close 하도록 만듬

- 이때 모든 표준 Spring lifecycle callback을 사용할 수 있음 (DisposableBean interface, @PreDestroy )

- SpringApplication.exit(0 호출 할 때 특정 종료코드 반환하도록 할 수 있음 =\> org.springframework.boot.ExitCodeGenerator interface 구현

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  @SpringBootApplication
  public class MyApplication{
    @Bean
    public ExitCodeGenerator exitCodeGenerator() {
      return () -> 42;
    }
    
    public static void main(String[] args) {
      System.exit(SpringApplication.exit(SpringApplication.run(MyApplication.class, args)));
    }
  }
  ```

  </div>

  </div>

- ExitCodeGenerator interface는 Exception 클래스에서 구현해도 됨 =\> ExitCodeGenerator interface를 구현한 Exception이 발생하면 종료코드 리턴

- ApplicationStartup은 SpringApplication 인스턴스 설정시 선택

  ``` syntaxhighlighter-pre
    }
  }
  ```

  </div>

  </div>

## CommandLineRunner

- Core Java 에서 Thread의 run() 메서드와 같은 기능을 테스트하는 동안 매우 편리한 Spring Boot 기능

- Spring Boot 애플리케이션이 시작된 후, 일부 특정 코드를 실행할 때 CommandLineRunner interface를 구현

- CommandLineRunner interface는 **SpringApplication.run()이 완료되기 직전에 호출**되는 단일 run() 메서드 제공

- 동일한 목적으로 **ApplicatoinRunner** interface를 사용할 수 있음

- 예제코드

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  @Component
  public class MyCommandLineRunner implements CommnadLineRunner {
    @Override
    public void run(String... args) {
      // code that ou want to run only once
    }
  }
  ```

  </div>

  </div>

- 특정 순서로 호출해야 하는 CommandLineRunner interface를 구현한 여러 개의 클래스나 ApplicationRunner interface를 구현한 클래스가 있을 경우, **@Order** annotation을 사용하거나 **org.springframework.core.Ordered** interface 를 구현하여 순서 지정

## Externalized Configuration

- 다른 환경에서 동일한 애플리케이션 코드로 작업할 수 있게 구성을 외부화할 수 있는 기능 제공

- Java Properties 파일, YAML 파일, 환경 변수 및 명령중 인수 등 다양한 외부 구성 소스를 사용

- 그 중에 .properties 와 .yml 두가지 형식의 파일이 Spring boot 에서 가장 많이 사용

- ‘.properties' 와 ‘.yml’ 파일이 동일한 위치에 있을 경우 ‘.properties’ 가 우선시 됨

### YAML

- YAML 은 JSON의 상위 집합, 계층적 구성 데이터를 지정하기 위해 편리한 형식

- 클래스 경로에 SnakeYAML 라이브러리가 있는 경우 Spring Boot는 ‘.properties’ 파일의 대안으로 YAML 을 자동으로 지원

- “Starters” 를 사용하면 자동으로 SnakeYAML을 spring-boot-starter와 함께 제공됨

- YAML 파일의 확장자는 ‘.yml’

## GraalVM Native Image 지원

- Spring Boot 3.0 release에 도입된 새로운 기능

- GraalVM Native Image 는 Java 애플리케이션을 배포하고 실행할 수 있는 새로운 방법 제공

- JVM과 비교할 때 Native Image는 더 작은 메모리 공간과 훨씬 빠른 시작시간으로 실행 가능

- GraalVM 은 Java 및 기타 JVM 언어로 작성된 애플리케이션의 실행 속도를 높임과 동시에 Javascript, Python 및 기타 여러 인기 언어에 대한 runtime을 제공하기 위한 고성능 JDK

- GraalVM은 Java 애플리케이션을 실행하는 두 가지 방법 제공

  - Graal JIT(Just-In-Time) 컴파일러가 있는 HotSpot JVM에서 실행

  - AOT(Ahead-Of-Time) 컴파일된 Native 실행파일로 실행

- GraalVM Native Image 는 컴파일된 Java 애플리케이션을 미리 처리하여 생성할 수 있는 독립 실행형 실행 파일 : <u>추가 자료 필요, 이 장의 내용으로는 정확한 의미나 개념 파악 힘듬</u>
