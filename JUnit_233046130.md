# Annotation을 사용한 테스트

Annotation

- @Test : 메소드 호출시 새로운 인스탄스가 생성되어 독립적인 test 수행

- @Ignore : test 무시 (진행 안함)

- @Before : @Test가 진행되기 전에 빈드시 실행, @Test 실행전 공통, 필수 정보 설정시 사용

- @After : @Test 실행된 후 실행

- @BeforeClass : @Test보다 먼저 한번만 수행하여야 할 경우

- @AfterClass : @Test 보다 나중에 한번만 수행

# 테스트 픽스쳐(test fixture)

- 테스트 픽스쳐 모델이 JUnit4에서 향상됨

## 픽스쳐를 사용하는 이유

- 특정 로직이 테스트 전후에 실행되도록 보장하는 하나의 약정

- 손쉽게 재활용 가능

- annotation을 통해 명시적으로 변경(이전 버전은 구현여부 상관없이 적용됨)

- 테스트 전후에 픽스쳐를 실행할 수 있도록 보장하는 약정을 통해 재사용 가능한 로직 코딩 가능

- 픽스쳐는 동일한 로직을 사용하는 여러 테스트를 실행중이고 이들 중 일부 또는 전체가 실패할 경우 유용

- 각 테스트의 설정 로직에서 실패 원인을 확인하는 대신 한 곳에서 실패 원인을 추론 가능

- 테스트가 일부가 성공하고 일부는 실패하는 경우 이러한 실패의 공통 원인으로 픽스쳐 로직을 검사하지 않아도 됨

## JUnit4 에서의 fixture 사용

- annotation을 사용하여 픽스쳐에 대한 오버헤드를 없앰

- 픽스쳐 주석은 클래스 수준의 2개와 메서드 수준의 2개가 존재

- 클래스는 @BeforeClass, @AfterClass, 메소드는 @Before, @After

- 예제(매 Test 메소드 전에 실행)

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  import java.util.regex.Matcher;
  import java.util.regex.Pattern;
   
  import org.junit.BeforeClass;
  import org.junit.Test;
  import static org.junit.Assert.assertTrue;
  import static org.junit.Assert.assertFalse;
   
  public class RegularExpressionJUnit4Test {
    private String zipRegEx = "^\\d{5}([\\-]\\d{4})?$";
    private Pattern pattern;
    
    @Before
    public static void setUpBeforeClass() throws Exception {
      pattern = Pattern.compile(zipRegEx);
    }
    
    @Test
    public void verifyZipCodeNoMatch() throws Exception {
      Matcher matcher = this.pattern.mathcer("22101");
      boolean notValid = matcher.matches();
      assertFalse("Pattern did validate ip code', notValid);
    } 
    
    @Test(expected=IndexOutOfBoundsException.class)
    public void verifyZipCodeGroupException() throws Exception {
      Matcher matcher = this.pattern.mathcer("22101-5051");
      boolean isValid = matcher.matches();
      matcher.group(2);
    } 
  }
  ```

### Categories runner

- Categories runner는 @IncludeCategory 어노테이션에 정의된 클래스, 메소드만 실행

- @ExcludeCategory 어노테이션은 제외할 클래스 정의

- 예제

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  public interface FastTests {}
  public interface SlowTests {}

  public class A {
    @Test
    public void a() {
      fail();
    }
    
    @Category(SlowTests.class)
    @Test
    public void b() {
    }
    
    @Category({SlowTests.class, FastTests.class})
    public class B {
      @Test
      public void c() {
      }
    }
    
    @RunWith(Categories.class)
    @IncludeCategory(SlowTests.class)
    @SuiteClasses({A.class, B.class})     //Note that Categories is a kind of Suite
    public class SlowTestSuite() {
      // Will run A.b and B.c, but not A.a
    }
    
    @RunWith(Categories.class)
    @IncludeCategory(SlowTests.class)
    @ExcludeCategory(FastTests.class)
    @SuiteClasses({A.class, B.class})     // Note that Categories is a kind of Suite
  ```

## 간단한 매개변수를 통한 테스트

1.  매개변수를 사용하지 않는 일반 테스트 작성

# Assert 정의

## assertEquals(\[String message\], expected, actual)

- 변수 혹은 객체 두개의 값이 같은지 검사

- 어떤 형의 객체이던 동일성 테스트

## assertSame(\[String message\], Object exptected, Object actual)

- 두 객체가 같은 인스턴스 인지 검사

## assertNotSame(\[String message\], Objecte expected, Object actual)

- 두 객체가 다른 인스턴스 인지 검사

## assertNull(\[String message\], Object object)

- 해당 객체가 Null 값인지 검사

## assertNotNull(\[String message\], Object object)

- 해당 객체가 Null이 아닌지 검사

## assertTrue(\[String message\], boolean condition)

- 조건문이 true인지 검사

## assertFalse(\[String message\], boolean condition)

- 조건문이 false인지 검사
