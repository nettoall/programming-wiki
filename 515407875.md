## Spring Boot 3.x: Spring Security 5.x

- 샘플 코드1

  <div class="code panel pdl" style="border-width: 1px;">

  ``` syntaxhighlighter-pre
  @Configuration
  @EnableWebSecurity
  @RequiredArgsConstructor
  public class SecurityConfig {
    private final CustomAuthenticationManager customAuthenticationManager;
    private final UserFindPort userFindPort;
    private final Environment environment;

    @Bean
    public WebSecurityCustomizer webSecurityCustomizer() {
      return (web) -> web.ignoring()
        .requestMatchers(new AntPathRequestMatcher("/h2-console/**"))
        .requestMatchers(new AntPathRequestMatcher("/favicon.ico"))
        .requestMatchers(new AntPathRequestMatcher("/css/**"))
        .requestMatchers(new AntPathRequestMatcher("/js/**"))
        .requestMatchers(new AntPathRequestMatcher("/img/**"))
        .requestMatchers(new AntPathRequestMatcher("/lib/**"));
    }

    @Bean
    protected SecurityFilterChain filterChain(HttpSecurity http, HandlerMappingIntrospector introspector) throws Exception {
      http.csrf(AbstractHttpConfigurer:disable);
      http.authorizeHttpRequests(authorize ->
                                authorize.requestMatchers(new MvcRequestMatcher(introspector, "/**")).permitAll()
                                .anyRequest().authenticated())
        .addFilter(getAuthenticationFilter())
        .httpBasic(Customizer.withDefaults());
      return http.build();
    }

    private AuthenticationFilter getAuthenticationFilter() {
      return new AuthenticationFilter(customAuthenticationManager, userFindPort, environment);
    }
  }
  ```

# Authorize HttpServletRequests

## HttpSecurity

- HttpSecurity 인스턴스를 사용하면 authorization rules 를 선언해야 함.

- authorizehttpRequest를 사용 (최소한 사용해서 정의해야 함)

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  http
    .authorizeHttpRequests((authorize) -> authorize
      .anyRequest().authenticated())
  ```

  </div>

  </div>

- AuthorizationFilter는 FilterSecurityInterceptor 를 대체 (Spring Security 5.x 이후 사용 방법? - 확인 필요)

- 이전 버전과의 호환성을 유지하기 위해 FilterSecurityInterceptor는 기본값으로 남김

- AuthorizationFilter 작동 방법와 기본 설정을 덮어 쓰는 방법

- AuthorizationFilter 는 HttpServletRequest에 대한 authorization을 제공

- Security Filters 등 중에 하나로 FilterChainProxy에 삽입됨

- SecurityFilterChain 을 정의할 때 기본으로 덮어씀

- authorizeRequests 를 사용하는 대신 authorizeHttpRequests 사용 함

  - metadata sources, 구성 요소, decision managers 그리고 voters를 사용하는 대신 간단한 AuthorizationManager API를 사용 : 재사용과 커스터마이징 간단하게 함

  - 인증 조회를 지연 : 모든 요청에 대해 요구되는 인증 대신, 인증을 요구하는 authorization decision이 있는 요청에 대해서만 조회

  - bean-based 설정 지원

- authorizeHttpRequests가 authorizeRequest 대신 사용되면 FilterSecurityInterceptor 대신 AuthorizationFilter 가 사용됨

## AuthorizationFilter

- AuthorizationFilter는 Spring Security filter chain 에서 마지막에 위치

- Spring MVC endpoints 는 DeipatcherServlet에 의해 실행되는데 AuthorizationFilter 다음에 옴

- endpoints는 허용되기 위해 authorizeHttpRequests 에 포함되어야 한다.

## All Dispatches are Authorized

- AuthorizationFilter는 모든 요청 뿐만 아니라 모든 dispatch에 실행됨(dispatch 보내다)

- REQUEST dispatch는 authorization이 필요하지만 FORWARDs, ERRORs 그리고 INCLUDEs 도 승인이 필요하다.

- 예제 Spring MVC는 요청을 Thymeleaf template을 렌더링하는 view resolver로 FORWARD할 수 있음

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre

    public String endpoint() {
      return "endpoing";
    }
  }
  ```

  </div>

  </div>

  - 위 예제에서 authorization은 두번 발생함. “/endpoint” 시 한번, “endpoint” 시 한번

  - 위와 같은 이유로 모든 FORWARD dispatches에 대해 authorization 허용

- Spring Boo 에러 처리 예제

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">


  - Spring Boot가 ERROR dispatch로 보냄

  - 앞서 FORWARD와 같이 두번의 authorization 발생 → 모든 ERROR dispatches에 대해 허용해야

### Authentication Lookup is Deferred

- 인증 조회는 지연됨(의미가?)

- AuthorizationManager API는 Supplier\<Authentication\> 을 사용

- 요청이 항상 허용되거나 항상 거부되어야 할 때 authorizeHttpReqests 에 중요하다 함. → 이런 경우 인증은 조회되지 않으므로 빠른 요청을 만든다.

## Request 매칭하기

- 지금까지 2가지 요청을 매칭하는 방법을 적용함

- 첫번째는 간단하고 모든 요청을 매칭하는 것이고

- 두번째는 URI 패턴으로 매칭. Spring Security는 URI Pattern 매칭을 하기 위해 Ant와 Regular Expressions를 사용

### Ant로 매칭

- Spring Security라 request를 매칭하는 기본 언어

- single endpoint나 디렉토리 매칭, 나중에 사용하기 위해 palceholders를 캡쳐할 수 있음

- 특정 HTTP method 셋에 매칭하도록 구체화할 수 있음

- /endpoint 를 매칭하는 대신 /resource 디렉토리 밑에 있는 모든 endpoint를 매칭시키는 예제

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  http
    .authorizeHttpRequests((authorize) -> authorize
                          .requestMatchers("/resource/**").hasAuthority("USER")
                          .anyRequest().authenticated()
                          )
  ```

  </div>

  </div>

- 요청이 /resource 나 하위 디렉토리 일 때는 USER 권한이 필요하고 나머지는 인증해야 함.

- 다음과 같이 풀어 쓸 수 있음

  <div class="code panel pdl" style="border-width: 1px;">

### Dispatcher Type 매칭하기

- Spring Securty는 모든 dispatcher type을 기본으로 승인함

- REQUEST dispatch에 의거하여 성립된 security context 가 후속 dispatches 로 이어질 때 미세한 불일치는 가끔 예상치 못한 AccessDeniedException의 원인이 됨

- FORWARD와 ERROR와 같은 dispatcher type을 허용하는 Spring Security java 설정을 다음과 같이 설정

  ``` syntaxhighlighter-pre
                          .anyRequest().denyAll())
  ```

### MvcRequestMatcher 사용하기

## Requests 승인하기

- request가 매칭되며 permitAll, denyAll 또는 hasAuthority와 같은 방법으로 승인할 수 있음

- DSL에서 승인 규칙

  - permitAll: 승인하지 않고 그냥 허용하거나, 세션으로부터 인증 안함

  - denyAll: 모두 거부, 인증 안함

  - hasAUthority: 설정한 권한과 일치하는 경우

  - hasRole : 역할, “ROLD\_” prefix 사용

  - hasAnyAuthority : 설정한 어떠한 권한이라도 요구

  - hasAnyRole : 어떤 역할이라도

  - access : 커스텀 AuthorizationManager가 결정한 접근 제어 사용

  - 예제

    <div class="code panel pdl" style="border-width: 1px;">

    <div class="codeContent panelContent pdl">

    ``` syntaxhighlighter-pre
    @Bean
    SecurityFilterChain web(HttpSecurity http) throws Exception {
      http
        .authorizeHttpRequests(authorize -> authorize
                              .dispatcherTypeMatchers(FORWARD, ERROR).permitAll()
                              .requestMatchers("/static/**", "/signup", "/about").permitAll()
                              .requestMatchers("/admin/**").hasRole("ADMIN")
                              .requestMatchers("/db/**").access(allOf(hasAuthority("db"), hasRole("ADMIN")))
                              .anyRequest().denyAll()
                              );

      return http.build();
    }
    ```

    </div>

    </div>

- dispatcherType FORWARD, ERROR 모두 허용

- /static/, /sighup , /about 모두 허용

- /admin endpoint는 ADMIN 역할로 접속 “ROLE\_”은 생략됨

- /db 와 매칭되는 경우, ADMIN 역할과 db 권한 요구

- 나머지는 거부

## SpEL로 승인 표현하기

## 표현식 변환히기

- Where possible, SpEL 대신 type-safe authorization mangers 사용을 추천

- legacy SpEl을 변환하기 위해 WebExpressionAuthorizationManger 사용가능

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  .requestMatchers("/test/**").access(new WebExpressionAuthorizationManager("hasRole('ADMIN') && hasRole('USER')")
  ```

  </div>

  </div>

- @WebSecurity.check(authentication, request) 대신 다음과 같이 직접 호출하는 것이 나음

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  .requestMatchers("/text/**").access((authentication, context) ->
                                     new AuthorizationDecision(webSecurity.check(authentication.get(), context.getRequest())))
  ```

  </div>

  </div>

- bean 참조나 기타 표현식을 포함하는 복잡한 지침이 있을 때 그것들을 AuthorizationManger를 구현하거나, .access(AuthorizationManager) 를 호출하여 참조하는 식으로 변경하는 것을 추천

- AuthorizationManager를 사용하지 못하는 경우 bean Resolver와 같이 (또는 bean Resolver로) DefaulthttpSecurityExpressionHandler를 설정하고 이는 WebExpressionAuthorizationManager.setExpressionhandler 를 제공할 수 있음

## Security Matchers

- RequestMatcher 인터페이스는 요청이 주어진 규칙과 일치하는지 결정하는 데 사용

- SecurityMatchers는 주어진 HttpSecurity가 주어진 요청에 적용되어야 하는지 결정할 때 사용

- RequestMatchers는 주어진 요청에 적용되어야 하는 authorization rules를 결정하는데 사용

- 예제

  <div class="code panel pdl" style="border-width: 1px;">

# WebSecurity vs HttpSecurity

## WebSecurity

- WebSecurity의 ignoring에 endpoint를 만들면 Security Filter Chain이 적용 안 됨 : 인증, 인가 서비스 적용 안 함.

- Security Context 설정 안 함

- CSS, XSS, content-sniffing에 취약

- 일반적으로 로그인 페이지, public 페잊, 인증, 인가가 필요하지 않는 endpint에 사용

- healthcheck endpoint 에도 적용하였으니 Spring 시작시 해당 endpint 적용 추천 안 하고 permitAll 사용하라 권장함

- 예제

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  @Override
  public void configure(WebSecurity web) throws Exception {
    web.ignoring().antMatchers("/health", "/health/**");
  }
  ```

  </div>

  </div>

- 예제 (Spring 6.x)

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  @Bean
  public WebSecurityCustomizer webSecurityCustomizer() {
    return (web) -> {
      web.ignoring().requestMatchers(PathRequest.toStaticResource().atCommonLocations());
      web.ignoring().antMatchers("favicon.ico", "/resources/**", "/error");
    };
  }
  ```

  </div>

  </div>

## HttpSecurity

- HttpSecurity의 permitAll에 endpoint 적용시 인증 처리 무시, Security Filter Chain에 적용됨

- CSS, XSS, content-sniffing 검증

- 예제

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  @Bean
  public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
      .authorizeRequest()
      .antMatchers("/user").hasRole("USER")
      .antMatchers("/manager").hasRole("MANAGER")
      .anyRequest().authenticated()
    .and()
      .formLogin();

    return http.build();
  }
  ```

  </div>

  </div>

- SecurityFilterChain 을 여러 개의 메소드로 나누어서 정의하더라도 httpSecurity로 정의한 모든 endping에 적용 됨

- 서버 시작시 콘솔 로그에서 이전 security 설정에서는 RequestMatcher를 사용하여 Will Secure or \[Ant\[pattern='/…' \] with와 같이 특정 endpoint에 특정 설정을 할 수 있었으나 Spring Security 6.x 부터는 RequestMatcher를 지원하지 않음 (예제 Spring Security 6.x 이전 사용하던 방식)

  <div class="code panel pdl" style="border-width: 1px;">

  <div class="codeContent panelContent pdl">

  ``` syntaxhighlighter-pre
  @Bean
  @Order(2)
  public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    ...
  }

  @Bean
  @Order(1)
  public SecurityFilterChain exceptionSecurityFilterChain(HttpSecurity http) throws Exception {
    ...
  }
    
  ```

  </div>

  </div>

- Spring Security 6.0부터는 SecurityFilterChain을 리턴하는 하나의 메소드만 정의해서 처리하는 것이 편리함

- WebSecurity를 처리하는 (정확히 말하면 web.ignorint() 을 처리하는) WebSecurityCustomizer 리턴하는 메소드는 사용할 유형을 좁혀서 사용하는 것이 나을 듯 (예로 static page나 인증/인가 등을 거치지 않는 URL 등), healthcheck를 수행하는 url은 ignoring을 적용하는 것보다 permitAll 을 적용하는 것을 추천
